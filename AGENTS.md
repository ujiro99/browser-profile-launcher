This file provides guidance to AI agents when working with code in this repository.

## What This App Does

**browser-profile-launcher** is a desktop app (built with [Wails](https://wails.io/)) that lists and launches Google Chrome and Microsoft Edge browser profiles. Users can search profiles by keyword, organize them into named collections, and configure launch options.

## Build & Development Commands

### Prerequisites

- Go 1.22+
- Node.js / npm
- Wails v2 CLI: `go install github.com/wailsapp/wails/v2/cmd/wails@latest`

### Common commands

```bash
# Run in dev mode (hot-reload for both Go and frontend)
wails dev

# Build production binary
wails build

# Frontend only (inside frontend/)
npm run dev       # Vite dev server
npm run build     # tsc + vite build

# Lint frontend (Biome)
cd frontend && npx biome check .
cd frontend && npx biome check --write .
```

There are no Go tests in this repo currently.

## Architecture

### Tech Stack

- **Backend**: Go, Wails v2 (WebView-based desktop framework)
- **Frontend**: React 18, TypeScript, Vite, Tailwind CSS, Radix UI, Biome (linter/formatter)
- **IPC**: Wails auto-generates TypeScript bindings in `frontend/wailsjs/go/` from Go methods bound on the `App` struct

### Go Backend (`main` package)

- `main.go` — Wails app entry point; also serves profile `.ico` files via a custom HTTP handler (`IcoLoader`)
- `app.go` — `App` struct with methods exposed to the frontend: `Run`, `List`, `LoadConfig`, `SaveConfig`, `GetVersion`. Also starts `watchProfiles` goroutine using `fsnotify` to detect browser profile changes and emit `profileChanged` events to the frontend.
- `config.go` — Reads/writes `config.json` via XDG config dir (`~/.config/browser-profile-launcher/config.json`). Falls back to old package name `chrome-profile-selector` for backward compat.
- `profile/` — Platform-specific profile discovery:
  - `profile.go` — Shared: parses Chrome/Edge `Local State` JSON files, returns `[]Profile`
  - `profile_darwin.go` — macOS paths and `open -n -a <browser> --args --profile-directory=<dir>`
  - `profile_windows.go` — Windows paths and `cmd /c start <browser> --profile-directory=<dir>`

### Frontend (`frontend/src/`)

- `main.tsx` → `App.tsx` — Root; uses `ProfilesContext` to pass profiles from Go down the tree
- **Services** (`services/config.ts`) — Singleton `Config` class that loads/saves config via Wails IPC, fires change listeners, and handles version migrations (1.3→1.4→1.5→current)
- **Hooks**:
  - `useConfig` — reads/writes the config singleton
  - `useProfile` — merges Go-sourced profiles with stored `ProfileDetail` (alias, collection membership, launch options)
  - `useHistory` — tracks recently launched profiles
  - `useCollection` — manages user-defined tab collections
- **Key types** (`services/config.ts`):
  - `ProfileDetail` — per-profile stored metadata: `key` (= `"${browser}-${directory}"`), collections, aliasName, launchOptions
  - `Collection` — `{ name, icon }` tab group
  - `LaunchOption` — CLI flag passed to the browser on launch
- `lib/utils.ts` — fuzzy search (Fuse.js), `profileKey`, list filtering/sorting, version comparison, `LaunchOption` → string conversion

### Wails IPC Bindings

Auto-generated files in `frontend/wailsjs/` — **do not edit manually**. Regenerated by `wails dev` or `wails generate module`. The `App` struct's exported methods become callable TypeScript functions (e.g., `Run(browser, directory, options)`).

### Config File Location

- macOS: `~/.config/browser-profile-launcher/config.json`
- Windows: `%APPDATA%\browser-profile-launcher\config.json`

### Profile Key Convention

Profiles are identified throughout the codebase by the string key `"${browser}-${directory}"` (e.g., `"Google Chrome-Profile 1"`). This key ties together Go-discovered profiles and frontend-stored `ProfileDetail` records.
